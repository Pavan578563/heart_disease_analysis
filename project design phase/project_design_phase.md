# Heart Disease Analysis - Project Design Phase

## 1. Overview

This phase focuses on translating the requirements for the Heart Disease Analysis project into a detailed design blueprint. It encompasses defining the system architecture, data flow, data preparation and analysis pipeline, visualization design principles, and the web integration strategy. The primary goal is to ensure that the project effectively leverages data visualization to provide actionable insights for various stakeholders.

## 2. System Architecture

The Heart Disease Analysis project employs a multi-component architecture designed to handle data processing, visualization, and web delivery:

*   **Data Source**: Raw heart disease data, including patient demographics, medical history, lifestyle choices, and clinical indicators. This data is expected to be stored in a database.
*   **Database Layer**: A relational database (e.g., SQL-based) for storing the collected raw data and potentially processed data. This layer facilitates efficient data retrieval and SQL operations.
*   **Data Preparation Layer**: Python scripts (`data_preparation.py`) are used for cleaning, transforming, and feature engineering the raw data. This layer ensures data quality and prepares it for visualization tools.
*   **Visualization Layer**: This layer has a dual approach:
    *   **Tableau Desktop**: The primary tool for creating interactive dashboards and data stories. It connects directly to the prepared data (or database) to build visualizations.
    *   **Plotly + Flask**: An alternative web-based visualization solution for reproducing Tableau's functionality. Plotly is used for generating interactive charts, and Flask serves these charts within a web application.
*   **Web Application Layer**: A Flask application (`app.py`) that serves as the interface for accessing the dashboards and stories. It integrates the visualizations (either directly from Tableau exports or dynamically generated by Plotly) and provides a user-friendly experience.
*   **Static Assets**: HTML templates (`templates/`), CSS, and JavaScript (`static/`) for the web application, ensuring a responsive and visually appealing interface.

## 3. Data Flow

The data flow within the Heart Disease Analysis system is structured as follows:

1.  **Data Collection**: Raw heart disease data is collected from various sources and ingested into the database.
2.  **SQL Operations**: SQL queries (`02_sql_operations.sql`) are performed on the database to extract, filter, and aggregate data as needed for analysis.
3.  **Data Extraction**: The relevant data is extracted from the database, potentially into CSV files (`data/`).
4.  **Data Preparation**: Python scripts (`data_prep/data_preparation.py`) process the extracted data. This involves:
    *   **Cleaning**: Handling missing values, correcting inconsistencies.
    *   **Transformation**: Reshaping data, creating new calculated fields.
    *   **Feature Engineering**: Deriving new features relevant for analysis (e.g., BMI categories).
5.  **Visualization Tool Connection**: The prepared data is then connected to either Tableau Desktop or used by Plotly within Python scripts (`visualizations/build_dashboards.py`).
6.  **Dashboard and Story Creation**: Interactive dashboards and data stories are designed in Tableau or programmatically generated using Plotly.
7.  **Web Integration**: The created dashboards and stories are integrated into the Flask web application:
    *   For Tableau, this might involve exporting static HTML files (`dashboard_clinician.html`, `dashboard_policy.html`, `dashboard_personal.html`, `story.html`) and embedding them.
    *   For Plotly, the Flask application dynamically renders the Plotly charts within Jinja templates.
8.  **User Access**: Users access the interactive dashboards and stories through a web browser via the Flask application.

## 4. Data Preparation and Analysis Pipeline

1.  **Problem Understanding**: Define the business problem and specific questions to be answered (e.g., identifying lifestyle factors, regional trends, personal risks).
2.  **Literature Survey**: Review existing research and best practices in heart disease analysis.
3.  **Data Collection**: Gather comprehensive datasets related to heart disease.
4.  **Data Storage**: Store data in a structured database, enabling efficient querying and management.
5.  **SQL Operations**: Utilize SQL for initial data exploration, aggregation, and creation of views for specific analytical needs.
6.  **Data Preparation**: Employ Python scripts for advanced data cleaning, transformation, and feature engineering to optimize data for visualization and analysis.
7.  **Visualization Development**: Create a variety of visualizations (e.g., bar charts, line graphs, scatter plots, heatmaps) to represent different aspects of heart disease data.
8.  **Dashboard Construction**: Assemble related visualizations into interactive dashboards, allowing users to filter, drill down, and explore data.
9.  **Storytelling**: Sequence dashboards and individual visualizations into a coherent data story to guide users through key insights and findings.

## 5. Visualization Design Principles

*   **Clarity and Simplicity**: Dashboards should be easy to understand, avoiding clutter and focusing on key metrics.
*   **Interactivity**: Incorporate filters, parameters, and drill-down capabilities to allow users to explore data dynamically.
*   **Persona-Centric Design**: Tailor dashboards to the specific needs and questions of each persona (cardiologist, policymaker, individual patient).
    *   **Clinician Dashboard**: Focus on patient-level details, risk factor correlations, and intervention effectiveness.
    *   **Policy Dashboard**: Emphasize regional trends, population health, and policy impact.
    *   **Personal Dashboard**: Provide simplified views of individual risk factors against benchmarks and actionable health advice.
*   **Responsiveness**: Ensure dashboards are viewable and functional across different devices and screen sizes.
*   **Aesthetics**: Use appropriate color schemes, fonts, and layouts to enhance readability and user engagement.

## 6. Web Integration Design

The web integration aims to make the analytical insights accessible to a broader audience without requiring specific software licenses (e.g., Tableau Desktop).

*   **Flask Application**: A lightweight Python web framework will host the dashboards.
*   **HTML Templates**: Jinja2 templates will be used to structure the web pages, embedding the visualizations.
*   **Static Files**: CSS and JavaScript will manage the styling and interactivity of the web interface.
*   **Dashboard Embedding**: The project will either:
    *   Embed static HTML exports from Tableau.
    *   Dynamically generate and embed Plotly charts within the Flask application, providing a fully open-source and interactive web experience.
*   **Routing**: Flask routes will be defined to serve different dashboards (e.g., `/clinician_dashboard`, `/policy_dashboard`, `/personal_dashboard`) and the data story (`/story`).
